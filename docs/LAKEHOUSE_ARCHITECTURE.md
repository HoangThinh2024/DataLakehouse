# Data Lakehouse Architecture

## Overview

This project implements a **proper Data Lakehouse** architecture ensuring all data transformations are versioned and stored in an immutable lake (RustFS). No analytics system queries source systems directly.

```
PostgreSQL Source → RustFS Bronze → RustFS Silver → RustFS Gold → ClickHouse Analytics
CSV Upload       → RustFS Bronze → RustFS Silver → RustFS Gold → ClickHouse Analytics
```

## Core Principles

### 1. **Lake-First Data Flow**
- All data is first extracted to **RustFS Bronze** (raw, unmodified)
- Transformations happen in **RustFS Silver** (cleaned, deduplicated, validated)
- Aggregations stored in **RustFS Gold** (analytics-ready, pre-computed)
- **ClickHouse reads only from RustFS**, never from PostgreSQL or CSV source

### 2. **Data Immutability**
- Every partition contains date-stamped files (e.g., `dt=2025-01-20/`)
- Every run has a unique `run_id` (UUID)
- Parquet format preserves schema and types across pipeline stages
- Files are never modified after creation → full versioning

### 3. **Recoverability**
- Any data can be regenerated by replaying from RustFS Bronze layer
- Silver/Gold layers can be recomputed without re-extracting
- ClickHouse tables can be rebuilt from RustFS if needed

### 4. **Lineage Tracking**
- Every row carries metadata:
  - `_pipeline_run_id`: UUID of the pipeline execution
  - `_source_table`: Original table name (e.g., `Demo`, `csv_upload`)
  - `_*.processed_at`: Timestamp when transformed
- Complete audit trail from source to ClickHouse

## Pipeline Architecture

### PostgreSQL → ClickHouse Pipeline

```
┌─ Extract Postgres ──────────────────────┐
│ Reads: PostgreSQL (public.*)            │
│ Output: In-memory DataFrame             │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Bronze to RustFS                        │
│ Writes: s3://bronze/demo/dt=X/UUID.parquet  │
│ (raw data, unmodified)                  │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Transform Silver                        │
│ - Drop duplicates                       │
│ - Validate & normalize data             │
│ - Parse types                           │
│ Output: cleaned DataFrame               │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Silver to RustFS                        │
│ Writes: s3://silver/demo/dt=X/UUID.parquet  │
│ (clean, deduplicated, typed)            │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Transform Gold                          │
│ - Daily aggregations                    │
│ - By-region summary                     │
│ - By-category summary                   │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Gold to RustFS                          │
│ Writes: s3://gold/{daily,region,cat}/ │
│         dt=X/UUID.parquet               │
│ (pre-computed analytics tables)         │
└────────────┬──────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│ Load to ClickHouse                      │
│ Reads: RustFS Silver & Gold (NOT In-Memory) │
│ - Reads latest Silver parquet files     │
│ - Reads latest Gold aggregation files   │
│ Inserts into:                           │
│ - analytics.silver_demo                 │
│ - analytics.gold_demo_daily             │
│ - analytics.gold_demo_by_region         │
│ - analytics.gold_demo_by_category       │
└────────────────────────────────────────┘
```

```
             │

└────────────┬───────────────────────────────┘
             │
┌────────────▼───────────────────────────────┐
│ CSV to RustFS Silver                       │

└────────────┬───────────────────────────────┘
             │

```

## Layer Structure in RustFS

### Bronze Layer (`s3://bronze/`)
- **Purpose**: Raw data capture from sources
- **Content**: Unmodified extracts
- **Retention**: Keep for 90 days (configurable)
- **Partitioning**: `{prefix}/dt={YYYY-MM-DD}/{run_id}.parquet`

```
bronze/
  demo/
    dt=2025-01-20/
      e4d2c1a0-{uuid}.parquet        (100k rows from PostgreSQL)
    dt=2025-01-19/
      f2c3b0a1-{uuid}.parquet        (100k rows from PostgreSQL)
  csv_upload/
    dt=2025-01-20/
      a1b2c3d4-{uuid}.parquet        (CSV file content)
```

### Silver Layer (`s3://silver/`)
- **Purpose**: Cleaned, validated data
- **Content**: Deduplicated, typed, normalized
- **Retention**: Keep indefinitely (source for analytics)
- **Partitioning**: `{prefix}/dt={YYYY-MM-DD}/{run_id}.parquet`

```
silver/
  demo/
    dt=2025-01-20/
      e4d2c1a0-{uuid}.parquet        (cleaned PostgreSQL data)
    dt=2025-01-19/
      f2c3b0a1-{uuid}.parquet
  csv_upload/
    dt=2025-01-20/
      a1b2c3d4-{uuid}.parquet        (cleaned CSV data)
```

### Gold Layer (`s3://gold/`)
- **Purpose**: Analytics-ready aggregations
- **Content**: Pre-computed metrics and dimensions
- **Retention**: Keep indefinitely (optimized for ClickHouse)
- **Partitioning**: `{aggregate_type}/dt={YYYY-MM-DD}/{run_id}.parquet`

```
gold/
  demo_daily/
    dt=2025-01-20/
      e4d2c1a0-{uuid}.parquet        (daily rollup: 1 row per day)
  demo_by_region/
    dt=2025-01-20/
      e4d2c1a0-{uuid}.parquet        (1 row per region)
  demo_by_category/
    dt=2025-01-20/
      e4d2c1a0-{uuid}.parquet        (1 row per category)
```

## Key Design Decisions

### 1. Why RustFS (S3-compatible)?
- **Immutable storage**: Parquet files never change after write
- **Affordable**: Object storage is cheaper than databases
- **Scalable**: Handles terabytes of data
- **Interoperable**: Standard S3 API works with tools (Spark, Presto, Pandas)

### 2. Why read ClickHouse from RustFS, not in-memory?
- **Compliance**: Data Lakehouse principle — lake is source of truth
- **Recoverability**: Can rebuild ClickHouse tables anytime
- **Auditability**: All transformations versioned in RustFS
- **Backpressure**: Prevents cascading failures (if ClickHouse down, data still safe in lake)

### 3. Why Parquet format?
- **Schema preservation**: Column types consistent across pipeline
- **Compression**: 5-10x smaller than CSV for numeric data
- **Fast reads**: Columnar format ideal for analytics queries
- **Tool support**: Pandas, Spark, DuckDB, Presto all support

## Metadata Fields

Every row in Silver/Gold layers includes:

| Field | Purpose | Example |
|-------|---------|---------|
| `_pipeline_run_id` | UUID of pipeline execution | `e4d2c1a0-...` |
| `_source_table` | Original table or source | `Demo`, `csv_upload` |
| `_extracted_at` | When extracted from source | `2025-01-20T15:30:00Z` |
| `_silver_processed_at` | When cleaned | `2025-01-20T15:31:00Z` |
| `_gold_processed_at` | When aggregated | `2025-01-20T15:32:00Z` |

These fields enable:
- ✓ Data lineage tracking
- ✓ Reprocessing past data
- ✓ SLA monitoring
- ✓ Duplicate detection

## Operations & Monitoring

### Check RustFS Layer Content

```bash
# List Bronze files
aws s3 ls s3://bronze/demo/ --recursive

# List latest Silver files
aws s3 ls s3://silver/demo/ --recursive | grep "dt=2025-01-20"

# Count Gold records
aws s3 ls s3://gold/demo_daily/ --recursive | wc -l
```

### Verify Data Integrity

```bash
python3 scripts/verify_lakehouse_architecture.py
```

This script checks:
- ✓ All layer buckets exist
- ✓ Data flows through Bronze → Silver → Gold
- ✓ ClickHouse tables populated from RustFS (not PostgreSQL)
- ✓ Pipeline metadata recorded

### Reprocess Yesterday's Data

```bash
# Read Bronze from specific date
python3 << 'EOF'
from mage.utils.rustfs_layer_reader import read_latest_layer
df = read_latest_layer('bronze', 'demo', date_str='2025-01-19')
# Transform and reload to ClickHouse
EOF
```

## Migration from Old Architecture

If you had:
```
PostgreSQL → (in-memory) → ClickHouse  ❌ NO LINEAGE
```

New architecture provides:
```
PostgreSQL → RustFS Bronze → RustFS Silver → RustFS Gold → ClickHouse  ✓ FULL LINEAGE
```

**Benefits:**
- Time-travel queries: "Which records changed between runs?"
- Replayability: Re-run any transformation without re-extracting
- Auditability: Every change tracked in lake storage
- Recoverability: Restore from any point in time

## Future Enhancements

Potential additions to the lakehouse:

1. **Delta Lake Integration**: Replace Parquet with Delta for ACID guarantees
2. **Schema Evolution**: Auto-detect schema changes and migrate
3. **Data Quality Framework**: DQ rules tracked in RustFS metadata
4. **Incremental Processing**: Only process changed rows (CDC from PostgreSQL)
5. **Access Control**: Lake-level permissions (who can read which tables)
6. **Time-Travel**: Query historical state of any table
7. **Streaming**: Real-time ingestion with exactly-once semantics
